# 项目架构红线

本项目（NPC AI 行为系统）专属禁令。通用禁令见 `docs/standards/red-lines.md`。

违反任何一条都意味着架构退化回 v1。

## 禁止硬编码业务逻辑

- **禁止**在 Go 代码中硬编码 FSM 状态定义或转换规则。必须从配置加载
- **禁止**在 Go 代码中硬编码事件类型与感知方式的映射。感知规则必须随事件配置声明
- **禁止**在 Go 代码中硬编码 NPC 类型特定的参数（速度、感知距离等）。必须从配置加载
- **禁止**用 switch-case 做 NPC 类型分发。必须用注册表/工厂模式

## 禁止破坏层次

- **禁止** BT 反向驱动 FSM。FSM 管"什么时候切状态"，BT 管"切了之后做什么"，方向单向
- **禁止** core/ 包 import runtime/ 或 gateway/。依赖方向必须单向向下
- **禁止** core/ 或 runtime/ 包 import experiment/。experiment/ 通过 build tag 隔离
- **禁止** gateway/ 承担非网络职责（AOI 计算、事件发布、Tick 调整等属于 runtime）

## 禁止弱类型通信

- **禁止** Blackboard 使用裸 `map[string]any`。必须通过泛型 `BBKey[T]` 访问
- **禁止** Blackboard Key 散落在各文件中。所有 Key 集中定义在 `keys.go`
- **禁止** Key 拼错静默失败。配置加载时必须校验引用的 Key 是否在注册表中
- **禁止** FSM 状态名使用魔法字符串。状态名必须有 const 约束

## 禁止实验污染核心

- **禁止**核心代码中出现实验相关的字段、方法或 import
- **禁止**为实验需求修改核心数据结构。实验通过外部组合核心组件实现

## 禁止实验作弊

### 公平性

- **禁止**故意削弱对照组实现来制造差距。每种模式必须是该架构约束下的最佳努力实现
- **禁止**在对照组中暗含实验组的核心能力

### 立论

- **禁止**把工业标配当创新点验证。实验要证明的是三层协作的架构模式价值
- **禁止**设计只能证明单一组件价值的实验。必须覆盖每一层的不可替代性

### 方法

- **禁止**先写代码再想验证。必须先定义命题和区分场景
- **禁止**对符合预期的数据直接采信。先攻击——改参数、极端输入、换场景
- **禁止**只用玩具规模验证架构优越性。必须在不同规模下测试并找交叉点
- **禁止**接受全零或全相同的指标数据。说明指标设计有问题，必须重新设计

## 禁止联调配置脱节

- **禁止** ADMIN 侧只修改本地文件就回复 READY。配置变更必须通过 ADMIN REST API 写入
- 服务端启动时从 `NPC_ADMIN_API` 全量拉取配置，ADMIN 未就绪则启动失败
