### 根因与代码修复

**Hybrid 单边回归的根源**：`Decision.Evaluate` 是 Hybrid 每 Tick 必经路径，PureFSM / PureBT 不走决策中心。自档案以来该路径有两处累积开销：

1. **slog.Debug 变长参数逃逸**（[internal/runtime/decision/decision.go:81](internal/runtime/decision/decision.go#L81)）：`slog.Debug("decision.evaluated", "npc_id", ..., 7 个 kv)` 在 info 级下虽被过滤，但 `...any` 参数包仍每 Tick 构造并逃逸到堆。
2. **三维评分 BB 写入**（#10 feature）：档案时代决策只写 1 个 BB（`KeyThreatLevel`），新版本写 4 个（`KeyThreatScore` / `KeyNeedScore` / `KeyEmotionScore` / `KeyDecisionWinner`）。即便默认权重下 Need/Emotion=0，写入本身的泛型 set 开销不可避免。

**本次修复**：第 1 项可消除，在热路径加 `slog.Default().Enabled(ctx, slog.LevelDebug)` 门控，回收 ~100 ns/Tick。第 2 项是特性语义，不改；残留 ~100 ns 是多维决策固有成本。

### 结论

- **架构三项核心结论仍成立**：
  1. Hybrid 综合最均衡（大规模 N≥100 比 PureFSM 快 87~154 ns，仍优于 PureFSM；PureBT 在中大规模增长快，N≈267 会追上 Hybrid 上限水平）
  2. Hybrid 配置边际成本为 0（图 3/4 确定性数据，未变）
  3. 三层各自不可替代（表 1 定性场景，全部 PASS 未变——本次也跑了 `TestExperiment_*`，4/4 通过）
- **小规模（N=10）处**，Hybrid 与 PureFSM 差距缩小到统计噪声内（Hybrid 843 vs PureFSM 874，仅 -3.5%）。答辩讲述时可说明："Hybrid 的三层开销在行为数少时相对被放大，交叉效益在 N ≥ 50 才稳定显现，这与 Hybrid 面向'大规模复杂行为集'的设计定位一致。"
- **图 7 综合评分雷达**需要基于刷新后的 N=100 档位重算（原数据仍可作对比），本次未重绘。

## 表 1：三层不可替代性（定性验证）

每行代表一个定性场景，每列代表一种架构模式。PASS = 行为正确，FAIL = 行为缺失。

| 场景 | 证明什么 | Hybrid | FSM+DC | BT+DC | PureFSM | PureBT |
|------|---------|--------|--------|-------|---------|--------|
| 距离陷阱 | DC 不可替代 | 100% | 100% | 100% | **50%** | **50%** |
| 多步骤行为 (current_action) | BT 不可替代 | **PASS** | FAIL | FAIL | FAIL | FAIL |
| 状态生命周期 (exit_cleanup) | FSM 不可替代 | **PASS** | **PASS** | FAIL | FAIL | FAIL |

**结论**：只有 Hybrid（FSM+BT+DC）全部 PASS，三层各自不可替代。

- **无 DC**（PureFSM/PureBT）：远处高 severity 事件未经距离衰减，threat=90 直接触发 Flee（误判率 50%）
- **无 BT**（FSM+DC/PureFSM）：current_action 始终为空，无法追踪多步骤行为进度
- **无 FSM**（BT+DC/PureBT）：无 OnExit 回调，状态退出时无法执行清理，脏数据残留
