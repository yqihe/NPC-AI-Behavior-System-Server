# /verify — 代码验证

验证代码是否正确。必须跑真实命令，最终判定 PASS 或 FAIL，不允许模糊结论。

## Usage
```
/verify <feature-name>
/verify <feature-name> --task=T1    # 只验证某个任务的改动
```

---

## 核心原则

**不是确认代码能工作，而是想方设法把它搞崩。**

如果你发现自己在想"这个应该不会出问题"——停下来，做相反的事，去试试能不能搞崩它。

---

## 执行流程

1. 加载 spec 的 requirements.md，提取验收标准（R1, R2, ...）
2. **在跑之前**，列出每条验收标准对应的具体验证方法和预期结果
3. 执行验证（必须跑真实命令）
4. 对每条标准判定 PASS / FAIL
5. 输出验证报告

---

## 验证手段（按优先级）

1. **`go test ./...`** — 单元测试全部通过
2. **`go test -race ./...`** — 并发代码无竞态
3. **`go build ./...`** — 编译通过
4. **`go test ./test/e2e/... -v`** — e2e 测试通过
5. **手动构造边界输入** — 空值、超大值、并发请求

---

## 禁止

- **不准只看代码就说通过**——必须跑真实命令，`go test` 输出是唯一证据
- **不准模糊结论**——每条验收标准必须判 PASS 或 FAIL，不存在"基本通过"、"应该没问题"
- **不准给自己找借口**——"这个测试不稳定所以跳过"不可接受
- **不准修改测试来迁就代码**——除非先确认服务端行为符合设计预期
- **不准跳过竞态检测**——涉及并发的代码必须跑 `-race`

## 攻击性测试思维

跑完基础测试后，主动尝试搞崩它：

- **nil 输入**：传 nil 参数，空 map，空 slice
- **并发压力**：多个 goroutine 同时操作
- **边界值**：0、负数、最大值、空字符串
- **时序问题**：事件乱序到达、超时、重复发送
- **配置缺失**：缺少必要的配置字段、Key 不存在

## 失败处理

断言 FAIL 时，**先定位是代码 bug 还是测试问题**：

1. 查服务端日志/输出，确认实际行为是什么
2. 对照 design.md，判断实际行为是否符合设计
3. **实际行为不符合设计** → 代码 bug → 建议 `/debug`
4. **实际行为符合设计但测试预期错误** → 修改测试
5. **实际行为符合设计但设计本身有问题** → 回到 spec 修改设计，需要用户审批

## 验证报告模板

```
## 验证报告

**Spec**：<feature-name>
**范围**：全部 / T[N]
**服务端版本**：[git log --oneline -1]

### 验收标准检查
| 标准 | 验证方法 | 预期 | 实际 | 判定 |
|------|----------|------|------|------|
| R1: xxx | go test -run TestXxx | PASS | PASS | PASS |
| R2: xxx | e2e 场景 A | 状态变为 Flee | 状态变为 Flee | PASS |

### 攻击性测试
| 测试项 | 结果 |
|--------|------|
| nil 输入 | PASS - 返回错误，不 panic |
| 并发读写 | PASS - race detector 无报告 |

### 结论
**PASS** / **FAIL**（X 条失败）

### 下一步
- PASS → 提交 commit，继续 `/spec-execute T[N+1] <feature-name>`
- FAIL → 跑 `/debug <symptom>`
```
