# v1 尸检报告

## 概况

v1 约 14,400 行 Go，开发 9 天，实现了 2/5 NPC 类型（平民、警察），68 个 e2e 断言全部通过，三模式对照实验 18/18 等价验证通过。

## 核心问题

**FSM 与 BT 的数据驱动程度不对等**：BT 树结构在 MongoDB 中配置驱动，FSM 转换规则硬编码在 Go 代码中。加一种 NPC 类型要改 11 个文件，其中 7 个因硬编码而改。

## 具体缺陷

| 缺陷 | 表现 | 根因 |
|------|------|------|
| FSM 硬编码 | `civilian.go`/`police.go` 用 Go 闭包写转换条件 | 没做配置化 |
| Blackboard 无 schema | `map[string]any`，Key 拼错不报错 | 无编译期校验 |
| NPC 类型 switch-case | `wireStates()`、`npcSpeed()` | 无注册表模式 |
| 事件→感知硬编码 | `canPerceive()` switch-case | 感知规则未配置化 |
| 实验侵入核心 | Instance import experiment/builders | 依赖方向反转 |
| 决策中心名不副实 | 只做事件→BB 写入，无威胁评估 | 后到事件覆盖先到 |
| Gateway 职责过重 | 路由+AOI+事件发布+Tick 调整混在一起 | 未拆分子系统 |
| 空间查询 O(n) | 每次事件遍历所有 NPC | 无空间索引 |

## 关键教训

1. **配置驱动必须对称**——BT 配置化了 FSM 也必须配置化，否则扩展时一半靠改配置一半靠改代码
2. **Blackboard 需要形式化契约**——无 schema 的 `map[string]any` 是隐形 bug 温床
3. **实验框架必须与核心隔离**——v1 的 ~1,100 行实验代码占总量 7.6%，且反转了依赖方向
4. **先纵向跑通再横向扩展**——v1 搭了太多基础设施（Admin 2,512 行、Redis 死代码），但只跑通 2/5 类型
