# experiment-layer 需求分析

## 动机

毕设核心创新点是 **FSM + BT + 决策中心 三层协作架构**——不是某个单独组件的创新（决策中心是工业标配），而是三者的职责划分和协作模式。

论文需要两类实验数据：

1. **定性验证**：三层各自不可替代——缺任何一层都会导致特定能力缺失
2. **定量验证**：随着行为复杂度和 NPC 规模增长，Hybrid 架构的综合优势逐渐显现——找到交叉点

### 论文叙事主线

1. 纯 FSM 的经典痛点：状态爆炸（状态数和转换规则随行为增多呈 O(n²) 增长）
2. 纯 BT 的经典痛点：树膨胀（一棵大树越来越深/宽，遍历耗时线性增长，状态管理混乱）
3. 传统做法是 FSM 或 BT 二选一 + 决策层，但各自的固有缺陷无法通过单独加决策层解决
4. 本系统让 FSM、BT、决策中心三者同时工作、各司其职，通过 Blackboard 解耦
5. 实验数据证明：小规模下三种架构差异不大，但随着规模增长，Hybrid 综合最优

### v1 教训

v1 实验代码侵入核心（依赖反转），v2 用 build tag 完全隔离。

## 优先级

**最高**。论文核心数据来源。

## 预期效果

### 第一部分：定性验证——三层不可替代性

#### 场景 A：距离陷阱——证明 DC 不可替代

远处高 severity 事件（gunshot at 290m, severity=90）。

- 有 DC 的模式：threat = 90×(1-290/300) = 3 → Alarmed（正确）
- 无 DC 的模式：threat = 90 → Flee（误判）

#### 场景 B：多步骤行为——证明 BT 不可替代

NPC 进入 Flee 状态后，应执行多步骤序列：assess_threat → plan_escape → run_away。

- 有 BT 的模式：BB 中 `current_action` 被正确设置为 "run_away"
- 无 BT 的模式：`current_action` 始终为空

#### 场景 C：状态生命周期——证明 FSM 不可替代

NPC 离开 Alarmed 状态时应清理 `alert_start_tick`。

- 有 FSM 的模式：OnExit 回调触发清理，`exit_cleanup_done` 被设置
- 无 FSM 的模式：无 OnExit 机制，脏数据残留

### 第二部分：定量验证——规模增长对比

#### 图 1：行为数量 vs 单 Tick 耗时（折线图）

x=行为数量(10,50,100,150,200)，y=单 NPC 单 Tick 耗时(ns)，3 条线（Hybrid/PureFSM/PureBT）。

**预期**：PureFSM 因转换规则 O(n²) 增长而加速上升；PureBT 因树节点线性增长而稳步上升；Hybrid 因 FSM 固定 ~5 状态 + BT 分治，增长率最低。找到交叉点。

#### 图 2：NPC 数量 × 行为复杂度 吞吐量矩阵（折线图组，4 子图）

每个子图对应一个行为档位(10/50/100/200)，x=NPC 数(100,500,1000,5000)，y=总 Tick 耗时(μs)，3 条线。

**预期**：行为少时三者差异小；行为多 + NPC 多时 Hybrid 优势明显。

#### 图 3：行为数量 vs 配置复杂度（折线图）

x=行为数量，y=转换规则数/树节点数/配置总行数。

**预期**：PureFSM 转换规则呈 O(n²) 增长；PureBT 单树节点数线性但可读性崩溃；Hybrid FSM 转换规则固定 + BT 分散在多个小文件中，O(n) 且可维护。

#### 图 4：边际扩展成本（分组柱形图）

在行为数量=10/50/100/200 处，各加 1 个新行为，记录改动行数。

**预期**：Hybrid 每次改动恒定（加一个 BT 子节点）；PureFSM 改动量随状态数增长（新增转换规则）；PureBT 改动量随树规模增长（在大树中插入分支）。

#### 图 5：行为数量 vs 内存/NPC（折线图）

x=行为数量，y=bytes/NPC。

**预期**：PureFSM 转换表内存随状态数 O(n²)；PureBT 树共享但节点状态存储线性增长；Hybrid 适中。

#### 图 6：行为数量 vs 单事件响应墙钟时间（折线图）

x=行为数量，y=从事件注入到 NPC 完成状态转换的墙钟耗时(ns)。

**预期**：PureFSM 需遍历大量转换规则，耗时增长；PureBT 需 DFS 遍历深树，耗时增长；Hybrid 固定少量 FSM 转换 + 小 BT 树，增长最慢。

#### 图 7：综合评分雷达图

在 100 行为 × 1000 NPC 规模下，5 个维度（Tick 耗时、响应速度、内存效率、配置复杂度、行为正确性），5 种模式各一条线。

**预期**：Hybrid 面积最均衡——不是每个维度都最优，但综合性价比最高。

## 五种模式定义（控制变量法）

| 模式 | FSM | BT | 决策中心 | 用途 |
|------|-----|-----|---------|------|
| **Hybrid** | ✓ | ✓ | ✓ | 完整架构（实验组） |
| **FSM+DC** | ✓ | ✗ | ✓ | 隔离 BT 的贡献 |
| **BT+DC** | ✗ | ✓ | ✓ | 隔离 FSM 的贡献 |
| **PureFSM** | ✓ | ✗ | ✗ | 隔离 DC 的贡献 |
| **PureBT** | ✗ | ✓ | ✗ | 隔离 DC 的贡献 |

**公平对比原则**：每种模式是该架构约束下的最佳努力实现，不人为削弱。差异来自架构固有限制。

## 行为数量的规模模拟方法

为了在 10/50/100/150/200 不同规模下测试，需要**程序化生成**对应规模的配置：

- **PureFSM**：N 个状态 + ≈N×(N-1)/2 条转换规则（每对状态之间可能有转换）
- **PureBT**：一棵大 Selector 树，N 个分支，每个分支 = Sequence(check + action)
- **Hybrid**：固定 5 个 FSM 状态 + 5 棵 BT 子树，每棵子树 ≈N/5 个行为节点

配置通过 Go 代码在测试中动态生成，不手写 JSON 文件。

## 不做什么

- 不做 WebSocket 或 Unity 集成
- 不做 NPC 间交互实验
- 不做自动化论文图表渲染（输出原始数据，手动用工具画图）
- 差异来自架构固有限制，不人为削弱对照组
- 不做手写 200 个行为的 JSON 配置——通过 Go 代码程序化生成
